# Praxis Agents — System Design

## Overview

A multi-agent platform where hierarchical manager agents dynamically plan and delegate tasks to executor agents. Manager agents receive requests, decide which executors to invoke and in what order (2-level dynamic planning), and synthesise results. Execution is asynchronous via BullMQ, with Notion as the knowledge and output layer.

## Understanding Summary

- **What:** Multi-agent platform with 2-level dynamic planning — manager agents plan execution at runtime, executor agents do the work
- **Why:** Automate content creation (blog, social, marketing) and planning workflows (project breakdown, sprint planning, strategy, task decomposition)
- **Who:** Internal team of 1–5 people, authenticated via EVM wallet (MetaMask/WalletConnect)
- **Key Constraints:**
  - TypeScript stack (frontend + backend)
  - Provider-agnostic LLM layer (swap OpenAI, Anthropic, local models)
  - Notion integration (read + write)
  - Self-hosted deployment
  - Async execution model
- **Non-Goals:** External client access, on-chain transactions beyond auth, real-time streaming, mobile app, multi-tenancy

## Assumptions

1. EVM wallet login is for identity/auth only — no token gating, payments, or on-chain state
2. Notion is the source of truth for company knowledge — agents read context and write deliverables back
3. The UI is a dashboard for launching workflows, reviewing results, and approving content — not a chat interface
4. Agent-to-agent communication happens server-side — users see the final result, not inter-agent conversation
5. Web UI is sufficient — no mobile app needed
6. Single company — no multi-tenancy or org-level isolation

## Agent Model

Two agent types, same underlying model:

### Manager Agents

Top-level domain agents (e.g. C-suite agents in the chat system). A manager agent:

1. Receives a user request and relevant context
2. Calls the LLM to generate an **execution plan** — an ordered list of executor agents to invoke, with extracted inputs for each
3. Dispatches executor steps sequentially via BullMQ
4. Receives executor results and synthesises a final response

Managers know which executors are available to them via a `delegatesTo` array on their agent definition. The plan is generated at runtime — not a static template.

### Executor Agents

Single-purpose agents that do one job well. An executor agent:

1. Receives typed input (from the manager's plan)
2. Calls the LLM, optionally uses tools
3. Returns a typed result

Executors never spawn sub-agents. They are the leaf nodes — recursion stops here (2-level maximum).

### Shared Properties

Every agent (manager or executor) has:

- **Role** — what the agent does
- **System prompt** — personality, constraints, output format
- **Tools** — available integrations/actions (Notion, etc.)
- **Input schema** — typed with Zod
- **Output schema** — typed with Zod

Agents are stateless functions — all state lives in workflow run records in Postgres. Agent definitions live in the database (editable via UI) with code-defined defaults.

### Manager Context Threading

Manager agents make two LLM calls per delegated run: one to plan, one to synthesise. These calls share a **persistent message thread** stored as a JSON column (`messages`) on the `execution_plans` row. The planning call's full conversation (system prompt, user message, Notion context, assistant planning response) is saved after planning. The synthesis call loads this thread and appends the executor results as a new user message, so the LLM sees one continuous conversation with a gap where executors did work. This keeps the manager's reasoning chain intact without re-explaining the user's intent.

## Orchestration

### Dynamic Planning (2-Level)

Manager agents generate execution plans at runtime. A plan is a structured array of steps:

```typescript
// Generated by the manager agent's LLM call — not a static template
{
  steps: [
    { agent: "researcher", input: { topic: "product launch trends" } },
    { agent: "writer", input: { brief: "{{researcher.output}}" } },
    { agent: "editor", input: { draft: "{{writer.output}}" }, canRequestRevision: true },
    { agent: "human_gate", input: { content: "{{editor.output}}" } }
  ]
}
```

The manager decides which executors to use, in what order, and what inputs each receives — based on the user's request, available executors, and Notion context. Different requests to the same manager produce different plans.

### Execution Flow

1. Request arrives (UI trigger or chat message)
2. Manager agent is selected (via router or explicit selection)
3. Manager calls LLM → generates execution plan (validated against available executors via Zod)
4. Plan + manager's LLM message thread persisted to `execution_plans` table
5. Orchestrator enqueues first step to BullMQ
6. Workers execute executor agents sequentially, store results, enqueue next step
7. `human_gate` steps pause for user approval
8. After all steps complete, manager's saved message thread is loaded, executor results appended, and manager synthesises final output in the same conversational context as the planning call

### Guardrails

- **Max steps per plan:** 8 (configurable). LLM-generated plans exceeding this are rejected and the manager is re-prompted with a tighter constraint.
- **Cost ceiling per run:** Configurable max token budget. Orchestrator tracks cumulative cost across all steps and aborts if exceeded.
- **Executor allowlist:** Manager can only reference executors listed in its `delegatesTo` array. Unknown agent slugs in a plan are rejected at validation time.
- **No recursive delegation:** Executor agents cannot generate sub-plans. The orchestrator enforces this — if an executor's output contains a plan structure, it's treated as data, not instructions.

## Authentication

1. User clicks "Connect Wallet" — MetaMask signs a SIWE message
2. Backend verifies signature, issues JWT in httpOnly cookie
3. Access control via wallet address allowlist in database
4. Simple `is_admin` flag for admin capabilities

## Notion Integration

Connected via OAuth. Agents access Notion through typed tools:

| Tool | Description |
|---|---|
| `notion.search` | Search pages/databases by keyword |
| `notion.readPage` | Read page content as markdown |
| `notion.readDatabase` | Query database with filters |
| `notion.createPage` | Create new page in specified parent |
| `notion.updatePage` | Update existing page properties/content |
| `notion.appendBlock` | Add content blocks to a page |

Agents **never delete** Notion content. All writes are logged for auditability.

## LLM Abstraction

Vercel AI SDK provides the unified interface. Each agent stores a `model` string (e.g. `"claude-sonnet"`) that maps to a concrete provider call at runtime via a single resolver function. No enum — adding a new model is one line in the resolver map + one option in the Agents UI dropdown.

### Model Registry

| Model Key | Provider | Concrete Model | Use Case |
|---|---|---|---|
| `claude-opus` | Anthropic | claude-opus-4-20250514 | Complex strategy, CEO/CFO synthesis |
| `claude-sonnet` | Anthropic | claude-sonnet-4-20250514 | Most managers + executors (default) |
| `gemini-flash` | Google | gemini-2.0-flash | Router, validation, task splitting |
| `gpt-4o-mini` | OpenAI | gpt-4o-mini | Cheap general-purpose fallback |
| `perplexity-sonar` | Perplexity | sonar-pro | Research with web search grounding |
| `deepseek` | DeepSeek | deepseek-chat | Code-related tasks |
| `grok` | xAI | grok-3 | Twitter/X data access |

### Resolver

```typescript
function resolveModel(model: string) {
  const models: Record<string, LanguageModel> = {
    'claude-opus':       anthropic('claude-opus-4-20250514'),
    'claude-sonnet':     anthropic('claude-sonnet-4-20250514'),
    'gemini-flash':      google('gemini-2.0-flash'),
    'gpt-4o-mini':       openai('gpt-4o-mini'),
    'perplexity-sonar':  perplexity('sonar-pro'),
    'deepseek':          deepseek('deepseek-chat'),
    'grok':              xai('grok-3'),
  }
  return models[model] ?? models['claude-sonnet']
}
```

### Special-Purpose Models

Perplexity and Grok are used as **tools** rather than general LLM brains. The researcher executor agent runs on Sonnet but calls Perplexity as a search tool. CMO social media executors use Grok as a Twitter data tool. The `model` field on these agents still controls the reasoning LLM — the special provider is accessed via the agent's tool configuration.

Fallback across providers if one is down. Every call logs tokens, model, cost estimate, and parent workflow step.

## Decision Log

| # | Decision | Alternatives Considered | Rationale |
|---|---|---|---|
| 1 | Custom monolith | LangGraph.js, Microservices | Simplest for small team, no framework lock-in, easy to self-host |
| 2 | TypeScript full stack | Python, Go | Team preference, unified language |
| 3 | Next.js App Router | Separate frontend + Express | Single deployable, SSR, co-located API |
| 4 | SIWE wallet auth | Privy, custom auth | Standard EVM auth, no third-party dependency |
| 5 | Vercel AI SDK | LangChain.js, direct API calls | Lightweight, multi-provider, TypeScript-native |
| 6 | BullMQ | pg-boss, Inngest | Mature, Redis-backed, simple API |
| 7 | Prisma ORM | Drizzle, Kysely | User preference, mature ecosystem |
| 8 | Custom orchestrator | LangGraph.js, Temporal | Full control, simpler mental model |
| 9 | Notion official SDK | Unofficial APIs | Direct integration, typed client |
| 10 | Agent defs in DB | Code-only | Edit prompts without redeployment |
| 11 | Docker Compose | K8s, cloud PaaS | Right complexity for 1–5 users |
| 12 | No Notion deletes | Full CRUD | Safety — avoid agent-caused data loss |
| 13 | 2-level dynamic planning | Static workflow templates; unbounded recursive planning | Static templates require manual definition for every new pattern. Unbounded recursion causes cost explosion, debugging nightmares, and non-determinism. 2-level (manager → executors) gives flexibility while staying debuggable. |
| 14 | Max 8 steps + cost ceiling per plan | No limits; per-step limits only | Hard cap prevents runaway plans. Cost ceiling catches cases where 8 cheap steps are fine but 8 expensive steps aren't. Both configurable per manager agent. |
